# Runbook — что делать, когда что-то сломалось

> Это инструкция для тебя в 3 утра, когда падает прод и приходят алерты.
> Шаги пишутся в **порядке от простого к сложному.** Не прыгай через пункты.

## 0. Первое действие при любом инциденте

1. **Не паниковать.** Большинство проблем решаются за 5–15 минут.
2. **Зафиксировать время начала** — пригодится для постмортема.
3. **Открыть status pages** в новой вкладке:
   - Supabase: https://status.supabase.com
   - Stripe: https://status.stripe.com
   - GitHub: https://www.githubstatus.com
   - Postmark: https://status.postmarkapp.com
   - Anthropic: https://status.anthropic.com
4. Если кто-то из них **down** — это не наша проблема. Подождать или твитнуть честно.
5. Создать файл `docs/incidents/YYYY-MM-DD-<short-name>.md` для постмортема — по ходу записывать что делал.

## Сценарий 1: Сайт не открывается / 404 / 500

### Диагностика

1. Открыть `https://finkley.app` в incognito браузере. Что видишь?
   - **404** — проблема с GitHub Pages или CNAME
   - **DNS error** — DNS не резолвится
   - **500** — ошибка приложения (но это маловероятно для статики)
   - **SSL error** — сертификат сломался

### Если 404

1. Проверить ветку `gh-pages` в GitHub репо — есть ли актуальный билд?
   - Если ветки нет или старая — посмотреть последний `Deploy Web` workflow в Actions
   - Если workflow упал — открыть логи, починить, перезапустить
2. Проверить `Settings → Pages`:
   - Source: должен быть `gh-pages` branch
   - Custom domain: `finkley.app`
   - Enforce HTTPS: включён
3. Проверить `apps/web/public/CNAME` — содержит ли `finkley.app`
4. Если всё ок, но 404 — попробовать `https://finkley.app/?p=dashboard` (SPA fallback). Если работает — проблема в `404.html`.

### Если DNS error

1. `dig finkley.app` или https://dnschecker.org — что возвращается?
2. Должно вести на GitHub Pages IP (`185.199.108.153` и т.п.) или CNAME на `<username>.github.io`
3. Если нет — открыть DNS-провайдера, проверить записи
4. DNS пропагация может занять до 48 часов — если только что менял, подожди

### Если SSL error

1. В `Settings → Pages` — проверить "Enforce HTTPS"
2. Иногда GitHub Pages SSL отваливается после изменения CNAME — нужно выключить-включить custom domain
3. Если не помогает — открыть GitHub Support ticket

## Сценарий 2: БД не отвечает / запросы падают

### Диагностика

1. Открыть Supabase status: https://status.supabase.com
2. Если **incident** на их стороне — ждать, твитнуть про "проблемы у нашего провайдера"
3. Открыть Supabase Dashboard → твой проект → Logs
4. Что в Postgres logs? Database paused? Connection limit exceeded?

### Если "Database paused"

Free tier Supabase усыпляет проект после 7 дней неактивности. Просто открыть проект в Dashboard — оно проснётся за 1-2 минуты.

### Если "Connection limit exceeded"

Free tier — 60 connections. Если упёрлись:

1. Проверить, не "залипли" ли долгие запросы (Logs → Postgres)
2. Перезапустить проект в Settings → Restart database
3. Если повторяется регулярно — пора на Supabase Pro ($25/мес, 200 connections)

### Если приложение падает на конкретном запросе

1. Открыть Sentry, найти ошибку
2. В Supabase Dashboard → SQL Editor — повторить запрос вручную
3. Если запрос падает с RLS-ошибкой — проблема в политике, не в коде. Проверить миграцию.

## Сценарий 3: Stripe webhook не доходит / подписки в кривом статусе

### Диагностика

1. Открыть Stripe Dashboard → Developers → Webhooks → твой endpoint
2. Раздел "Recent events" — приходят ли события? Какие ответы (200, 400, 500)?

### Если события приходят, но 400

1. Скорее всего — подпись webhook не сходится
2. Проверить `STRIPE_WEBHOOK_SECRET` в Supabase Function Secrets:
   ```bash
   pnpm supabase secrets list
   ```
3. Сравнить с "Signing secret" в Stripe Webhook settings
4. Если не совпадает — `pnpm supabase secrets set STRIPE_WEBHOOK_SECRET=whsec_...`

### Если события не доходят

1. Проверить URL endpoint в Stripe — правильный ли ref проекта Supabase?
2. Проверить, что edge function `stripe-webhook` задеплоена: `pnpm supabase functions list`
3. Если функции нет — `pnpm supabase functions deploy stripe-webhook --no-verify-jwt`

### Подписка в "incomplete" статусе

1. Открыть Stripe Dashboard → Customers → найти юзера → Subscription
2. Что в "events"? Возможно, карта декларнула 3DS
3. Юзер должен открыть Customer Portal и пере-аутентифицировать платёж
4. Кнопка в портале сработает автоматически

### Юзер заплатил, но в нашей БД status='trialing'

1. Найти `customer.subscription.updated` событие в Stripe
2. В Stripe → Replay этого события через UI
3. Если webhook теперь доходит — БД обновится. Если нет — см. "Если события приходят, но 400"

## Сценарий 4: Email не доходит до юзеров

### Диагностика

1. Открыть Postmark Dashboard → Activity
2. Видишь ли свежие отправки?

### Если отправок нет

1. Проблема в нашем коде — edge function `send-email` не вызвалась
2. Открыть Sentry, найти ошибку
3. Проверить `POSTMARK_SERVER_TOKEN` в Supabase secrets

### Если отправки есть, но статус "Bounced"

1. Email юзера невалиден — это не наша проблема
2. Если массовые bounces — проверить DKIM/SPF на mxtoolbox.com

### Если отправки есть, статус "Sent", но юзер не получил

1. Проверить SPAM папку юзера
2. Если в спаме — DKIM/SPF/DMARC проверка нужна (mail-tester.com)
3. Postmark может предупредить если репутация домена упала

### Если отправки есть, но в "Suppressions" список

Юзер в hard bounce list или manual suppression. Зайти в Postmark → Suppressions → удалить вручную если знаешь что email валидный.

## Сценарий 5: Утечка данных / подозрение на утечку

### Действия в первые 30 минут

1. **Не паниковать, не удалять никаких данных** — это нужно для расследования
2. Зафиксировать что именно случилось:
   - Логи доступа (Supabase Dashboard → Logs)
   - Sentry за последние 24 часа
   - GitHub Audit log (Settings → Audit log)
3. **Сменить пароли** на всех админских аккаунтах:
   - Supabase
   - GitHub
   - Stripe
   - Postmark
4. **Ротировать ключи:**
   - Supabase service-role-key (через Dashboard → API → reset)
   - Stripe live keys (Dashboard → Developers → API keys → roll)
   - Postmark Server Token
   - Anthropic API key

### Если ротируется SECRETS_ENCRYPTION_KEY

⚠ **Очень осторожно.** Это сломает все integration_credentials в БД.

1. Только если есть подтверждённая утечка ключа
2. Алгоритм: расшифровать всё старым ключом → зашифровать новым → сохранить
3. Это нетривиально и требует написания скрипта миграции
4. Альтернатива: удалить все integration_credentials, юзеры переподключат интеграции (выбрать как меньшее зло)

### GDPR-уведомление (Art. 33)

Если есть **подтверждённая** утечка персональных данных пользователей — уведомить **GDPR Authority в течение 72 часов**:

- В Польше: UODO (Urząd Ochrony Danych Osobowych) — uodo.gov.pl
- Параллельно — email затронутым пользователям

Постмортем — обязательно, в `docs/incidents/`.

## Сценарий 6: Edge function падает (500 / timeout)

### Диагностика

1. Supabase Dashboard → Functions → твоя функция → Logs
2. Что в логах? Stack trace?

### Типичные проблемы

**Timeout (10 сек на free tier, 60 сек на Pro):**

- Долгий запрос к внешнему API → добавить timeout на нашей стороне
- N+1 запрос к БД → переделать в один запрос или RPC

**`OutOfMemoryError`:**

- Загружаем огромный файл в память → стримить
- Накапливаем большой массив → пагинировать

**Cold start медленный:**

- Free tier — нормально (несколько секунд)
- Если критично — кеш на клиенте (TanStack Query) маскирует

### Перезапуск

Edge functions stateless, перезапуск не нужен. Просто пересобрать и задеплоить:

```bash
pnpm supabase functions deploy <name>
```

## Сценарий 7: Sentry молчит — приложение точно работает?

Иногда Sentry перестаёт получать события. Проверить:

1. В браузере открыть Network tab, обновить страницу
2. Должен быть запрос на `*.ingest.sentry.io` — есть ли?
3. Если нет — проверить `VITE_SENTRY_DSN` в env при последнем билде GitHub Actions
4. Если есть, но в Sentry Dashboard ничего — проверить "Project Settings → Client Keys" — DSN не отозван?

Это не критично — это инструмент мониторинга, не сам продукт.

## Сценарий 8: GitHub Actions сборка падает

### Диагностика

1. Actions tab в репо → последний failed workflow → логи

### Типичные проблемы

**`pnpm install` падает:**

- `pnpm-lock.yaml` рассинхронизирован — сделать `pnpm install` локально, закоммитить lockfile
- Версия pnpm не совпадает — проверить `package.json` поле `packageManager`

**`pnpm typecheck` или `pnpm lint` падает:**

- Локально проверить что зелёные перед push: `pnpm typecheck && pnpm lint`
- Если работает локально — версия Node.js разная? Проверить `.github/workflows/ci.yml` — `node-version: 20`

**Build падает с "missing env":**

- В Actions secrets не задан какой-то VITE\_\*
- Settings → Secrets → проверить список против `.env.example`

**Деплой gh-pages не происходит:**

- Проверить permissions в workflow: `permissions: contents: write`
- Проверить что branch `gh-pages` существует и не защищён правилом

## Сценарий 9: Я не могу работать сегодня (болезнь / личное)

Это тоже инцидент. Действия:

1. Поставить **status update** в Telegram-канал: "Сегодня не на связи, отвечу завтра"
2. Включить **auto-reply на support email**: "Спасибо за обращение, отвечу в течение 48 часов"
3. **Не пытаться "быстренько"** разрулить критичный инцидент в плохом состоянии — будет хуже
4. Если что-то горит и пользователи теряют деньги — твитнуть честно "знаем о проблеме, чиним к завтра"

## Сценарий 10: Claude Code зашёл в тупик / зацикливается / не знает что делать

### Симптомы

- Агент повторно делает одно и то же действие (читает один и тот же файл несколько раз)
- Агент пишет код, который уже был отвергнут — потом снова такой же
- Агент теряет нить разговора, не помнит обсуждённое решение
- Агент даёт расплывчатые ответы вместо конкретных действий
- Агент признаётся "не понимаю что делать дальше"

### Диагностика

1. **Сколько токенов в контексте?** Если близко к лимиту — агент захлёбывается
2. **Сколько часов идёт сессия?** После 4-6 часов качество падает
3. **Понятна ли задача?** Ты сам можешь объяснить её одним предложением?
4. **Есть ли свежий typecheck/lint в логах?** Может агент не знает что сломал

### Действия в порядке

#### 1. Короткий промпт-перезапуск (попробовать первым)

```
Стоп. Кратко в 3 пунктах:
1. Какая текущая задача (TASK-XX)?
2. Что ты сделал последним?
3. Что осталось?

Не пиши код. Только ответь на вопросы.
```

Если ответ внятный — продолжай. Если путаный — переходи к шагу 2.

#### 2. Запустить Definition of Done проверку

```
Запусти: pnpm typecheck && pnpm lint && pnpm test && pnpm build
Покажи выход. Не пиши код пока не покажешь.
```

Часто агент думает что что-то сделал, а на самом деле typecheck красный. Это сразу вернёт его в реальность.

#### 3. Принудительный refresh контекста

```
Стоп. Перечитай:
1. CLAUDE.md
2. Текущую задачу из docs/04_BACKLOG.md
3. Последние 3 коммита (git log -3 --stat)

Кратко: где мы и что делаем дальше.
```

#### 4. Делегировать sub-agent (если твоя версия Claude Code поддерживает)

Если задача стала большой и контекст загрязнён:

```
Передай задачу sub-agent reviewer для холодного взгляда.
Что он скажет про текущее состояние?
```

или

```
Используй sub-agent architect — нужно решение по архитектуре X.
Не пиши код, только ADR.
```

#### 5. Полный рестарт сессии

Если ничего не помогает:

- Выйти из Claude Code
- Зафиксировать прогресс в `docs/RETRO.md` или просто в новом файле `docs/checkpoint.md` (что сделано, что осталось, какие открытые вопросы)
- Перезапустить Claude Code
- Промпт начать с: «Прочитай `CLAUDE.md`, `docs/04_BACKLOG.md` (TASK-XX) и `docs/checkpoint.md`. Продолжи с того места где предыдущая сессия остановилась.»

#### 6. Если совсем не идёт

Возможно задача недостаточно атомарна. Проверь:

- Acceptance criteria понятны?
- Все блокирующие зависимости (TASK-XX) выполнены?
- Это XL-задача замаскированная под M? Разбей на подзадачи в backlog.

### Что НЕ делать

- ❌ Не давить «давай быстрее, ты уже 2 часа возишься»
- ❌ Не писать длинные многоступенчатые промпты после того как агент уже захлебнулся
- ❌ Не ругаться — это не помогает, только засоряет контекст
- ❌ Не молча включать-выключать Claude Code — теряется прогресс
- ❌ Не пытаться «исправить» промптом то, что нужно исправлять кодом

### Профилактика

- Сессии короткими сегментами (1-2 часа)
- В начале каждой сессии — `pnpm typecheck && pnpm lint && pnpm test` зелёные
- Между задачами — коммит и `git status` чистый
- Каждую неделю — ретро в `docs/RETRO.md`

## Сценарий 11: Claude Code сделал что-то деструктивное

### Что считаем деструктивным

- Удалил файлы, которые трогать было нельзя
- Изменил применённую миграцию
- Сделал push --force (не должен — есть branch protection и .claude/settings.json)
- Закоммитил .env с секретами
- Удалил данные в БД через `db:reset` на проде

### Действия

#### 1. Не паниковать

Большинство деструктивных действий обратимо через `git reflog`.

#### 2. Восстановление через git

```bash
git reflog                # история всех изменений HEAD
git reset --hard HEAD@{2} # вернуться на 2 действия назад
```

`git reflog` хранит ВСЁ за последние 90 дней, даже удалённые ветки. Спокойно.

#### 3. Если закоммичены секреты

```bash
# Сразу ротировать ВСЕ секреты, которые могли утечь
# Удалить из истории git:
git rm --cached <file>
git commit -m "chore: remove leaked secret"

# Если уже запушено — переписать историю и force-push
# (требует временного отключения branch protection)
```

И **обязательно ротировать ключи** в Supabase, Stripe, Postmark и т.д.

#### 4. Если повредил миграции

`supabase/migrations/` файлы хранятся в git. `git checkout HEAD~1 -- supabase/migrations/` вернёт.

Если миграция уже применена на staging/prod — нужна **корректирующая миграция**, не редактирование старой.

#### 5. Если потерял БД (db:reset на проде)

⚠ Это серьёзный инцидент.

- Открыть Supabase Dashboard → Database → Backups
- Восстановить из последнего автобэкапа (free tier — 7 дней retention, Pro — 14 дней)
- Все изменения после бэкапа потеряны
- Уведомить пользователей по email
- Постмортем в `docs/incidents/`

### Профилактика

- Не давай Claude Code прямой доступ к prod Supabase через MCP
- Branch protection на main
- `.claude/settings.json` с deny-листом для опасных команд
- Husky pre-push с typecheck + tests
- CI на PR

---

- **Supabase Discord** — обычно отвечают быстро на простые вопросы
- **Stripe Support** — приоритет для платящих, но free tier тоже отвечает
- **GitHub Support** — для проблем с GitHub Pages
- **Юрист по GDPR (на случай утечки)** — иметь контакт заранее, не искать в момент инцидента
