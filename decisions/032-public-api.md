# ADR-032: Публичный REST API салона (public-api)

## Статус

`Accepted`

Дата: 2026-06-15

## Контекст

В TASK-41 (8 мая 2026) сделали управление API-ключами: таблица `api_keys`
(SHA-256 хэш, scopes read/write), Edge Function `api-keys-create`
(генерит `fnk_live_…`), карточка в Настройки → API. Но **самого API не было**:
ни один эндпоинт не проверял эти ключи и не отдавал по ним данные. Ключи
создавались и вели «в никуда»; страница `Настройки → API` ссылалась на
несуществующий роут `/docs/api` и показывала curl на несуществующий домен
`https://api.finkley.app/v1`. С тех пор продукт оброс десятками новых сущностей
(склад, банк, контрагенты, кассы, повторяющиеся визиты, аналитика, …), а «API»
оставался заглушкой.

Владелец попросил: сделать **целостный** API (чтобы работали все основные
функции продукта, а не огрызок со старта), полную **публичную** документацию
(доступную без логина) с переходом из `Настройки → API`, и проверить
работоспособность. Права ключей — **read/write**.

## Решение

Построили публичный REST API как одну Edge Function `public-api` с
**декларативным реестром ресурсов** — единым источником истины.

- **Базовый URL:** `<SUPABASE_URL>/functions/v1/public-api`, версия путей `/v1`.
  Своего домена (`api.finkley.app`) не заводим — это платный Supabase custom
  domain; URL функции работает сразу.
- **Аутентификация:** `Authorization: Bearer fnk_live_…` или `x-api-key`.
  Ключ хешируется (SHA-256) и ищется в `api_keys` (revoked_at IS NULL).
  `verify_jwt = false` в `config.toml` — gateway не требует Supabase-JWT,
  потому что аутентификация делается нашим ключом внутри функции.
- **Scopes:** `read` → GET, `write` → POST/PATCH/DELETE. Нет права → 403.
- **Реестр** (`registry.ts`): для каждого ресурса заданы таблица/вью/RPC,
  читаемые колонки, колонки для create/update, фильтры, диапазон дат, money-поля
  (копейки), обязательные поля, правило скоупинга по салону. И сама функция, и
  страница документации читают этот реестр (через discovery-эндпоинт
  `GET /v1`), поэтому **документация физически не может разойтись с API**.
- **Изоляция салона** (service-role обходит RLS, поэтому скоуп — на нас):
  - прямые таблицы — `.eq(salon_column, salonId)`; на create `salon_id`
    проставляется сервером, клиент его задать не может (`pickAllowed` отбрасывает
    любые не-whitelisted поля);
  - parent-scoped таблицы без своей колонки салона (service-materials,
    staff-service-overrides, payout-lines, competitor-snapshots) — read-only,
    требуют проверенный `?<parent>_id=` (родитель должен принадлежать салону);
  - банк (bank-accounts/bank-transactions) — read-only, скоуп через
    `bank_connections.salon_id`;
  - RPC-аналитика — `p_salon_id` берётся из ключа, не из query.
- **Покрытие:** 46 ресурсов — визиты, доходы/категории, расходы/категории,
  платёжный календарь, клиенты, отзывы (вкл. Google/Booksy по ?source=),
  услуги/категории, мастера (с avatar_url), услуги мастера (staff-services),
  метрики салона (salon-metrics: рейтинг Google/Booksy), выплаты, склад,
  контрагенты, конкуренты, профиль салона, праздники, база знаний,
  касса (read), банк (read) + 12 аналитических RPC.
- **2026-06-15:** добавлена таблица `staff_services` (какие услуги выполняет
  мастер) + UI выбора в Справочники → Мастера; выставлена как ресурс
  `staff-services`. CORS дополнен `apikey`; discovery-fetch на странице доков
  сделан простым GET (без preflight). Бренд в доке — Finkley.
- **Документация:** публичный роут `/docs/api` (вне `RequireAuth`), читает живой
  каталог из `GET /v1`. Переход — кнопка в `Настройки → API`.

## Альтернативы, которые рассматривали

- **Хардкод эндпоинтов по одному** — отклонён: дублирует схему, разойдётся с
  продуктом (та самая проблема «API со старта устарел»). Реестр + discovery
  решают это by design.
- **Открыть Supabase PostgREST (`/rest/v1`) напрямую** — отклонён: требует
  Supabase-JWT (наши `fnk_live_` ключи туда не подходят), и пришлось бы
  раздавать anon-ключ + полагаться только на RLS для внешних клиентов.
- **Только документация без бэкенда** — отклонён: документировать
  несуществующий API нечестно, «проверить работоспособность» было бы нечего.
- **Custom-домен `api.finkley.app`** — отложен: платный, требует DNS; URL
  функции достаточен для MVP.

## Последствия

### Положительные

- Ключи `fnk_live_…` наконец работают; внешние интеграции (Zapier/n8n/Make/
  скрипты) получают доступ к данным салона.
- Документация всегда актуальна (генерится из того же реестра).
- Добавить новый ресурс при росте продукта = одна запись в `registry.ts`.
- Жёсткая изоляция салона и whitelist полей; покрыто unit/registry/smoke
  тестами (`deno test`) и адверсариальным security-review.

### Отрицательные / границы v1

- **Read-only в v1:** касса (cash-shifts/cash-transfers), движения склада,
  банк, parent-scoped ресурсы, выплаты (кроме status/paid_at/comment). Причина:
  их мутации идут через RPC с проверкой инвариантов (баланс кассы, склад) —
  прямой insert ломал бы консистентность.
- **3 аналитических RPC исключены** (`month_forecast`,
  `staff_performance_advanced`, `staff_tips_summary`): содержат
  `auth.uid()`-guard → при вызове service-role всегда падали бы с `forbidden`.
  Доступны только в приложении.
- **FK не проверяются на принадлежность салону** при create (например visit с
  чужим client_id): нельзя прочитать чужие данные, но можно создать «висячую»
  ссылку в своём салоне. Принято как ограничение v1.
- **Жёсткого rate-limit нет** — ограничение нагрузки через cap пагинации
  (max 200) + платформенные лимиты Supabase. Per-key throttling — на будущее.
- `media_posts` (глобальный блог) намеренно НЕ в API — это контент платформы,
  не данные салона.

### Что мониторим

- Если внешние клиенты начнут злоупотреблять — добавить per-key rate-limit
  (таблица счётчиков + окно).
- Если понадобится write для кассы/склада/банка — делать через выделенные
  RPC-эндпоинты, а не generic insert.
- Если аналитические RPC начнут возвращать пусто на проде — проверить
  EXECUTE-гранты для service_role (security invoker + дефолтный PUBLIC grant).
- Если заведём `api.finkley.app` — отдельный ADR (custom domain, стоимость).
