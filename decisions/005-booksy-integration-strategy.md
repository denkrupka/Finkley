# ADR-005: Стратегия Booksy-интеграции — открытый research-спайк

## Статус

`Proposed` · 2026-05-05 (в стадии 3 — `Accepted` после research-спайка TASK-27)

## Контекст

Booksy — главный конкурент-комплемент. Логика:

- Booksy = записи и календарь. Хорошо умеет.
- Finkley = прибыль. Хорошо умеет.

Большинство наших целевых юзеров уже сидят в Booksy. Если мы не интегрируемся — каждый юзер вручную вводит визиты дважды (в Booksy и у нас). Это болевая точка, и одновременно конкурентное преимущество если мы решим её.

**Проблема:** у Booksy нет публичного API для third-party приложений. Их API доступен только через Booksy Business app или через активную сессию салона. Login требует JS-rendering и hCaptcha.

**Стандартные подходы:**

1. **Платный прокси-сервис** (ScraperAPI, ScrapingBee, Bright Data) — $49+/мес. Отклонено владельцем (требование free tier).
2. **VPS с Playwright** — отклонено владельцем (нет VPS, не хочет администрировать).
3. **Бесплатный паттерн прокси-логина**, который владелец использует на другом своём проекте (sasovsky). Конкретная реализация — не знаю, владелец упомянул "API Scrapper, по-моему, мы подключались".

## Решение

**В стадии 1–2 — Booksy не делаем.** CSV-импорт работает с дня 1 как fallback (юзер выгружает CSV из Booksy раз в неделю и заливает к нам).

**В стадии 3** — research-спайк (TASK-27, оценка M):

1. Изучить рабочий паттерн прокси-логина владельца на sasovsky
2. Проверить, работает ли он на Booksy (там JS-rendering + hCaptcha, более сложный кейс чем строительный портал)
3. Принять одно из решений:
   - **A:** работает — адаптируем, делаем интеграцию (TASK-28..29)
   - **B:** не работает — делаем CSV-импорт продвинутым (smart-маппинг, шаблоны), без авто-синка
   - **C:** не работает, но критично для растущих клиентов — обсудить с владельцем платный прокси-сервис (~$49/мес → перенести в costs of revenue, окупается с 4–5 платных юзеров)

После решения — этот ADR переходит в `Accepted` с конкретным выбором.

## Альтернативы (отклонены сейчас, могут вернуться)

- **Booksy Public API через партнёрство.** Отклонено сейчас: у Booksy нет публичной программы, а мы — конкурент в части аналитики. Можно обратиться когда будем 1000+ юзеров.
- **Browser extension вместо backend-логина.** Технически интересно (юзер устанавливает chrome extension, она читает Booksy с его сессией, отправляет к нам). Отклонено сейчас: Chrome Web Store review для финансового приложения = месяцы. Может вернуться в V3.

## Последствия

### Положительные (от текущего решения)

- Стадии 1–2 не блокируются Booksy-вопросом
- CSV-импорт работает для всех источников (Booksy, Fresha, Treatwell, ручные таблицы)
- Не тратим деньги на платный прокси до фактического спроса
- Гибкость: после research-спайка можем выбрать оптимальный путь

### Отрицательные

- Без auto-sync Booksy может снизить activation rate (юзеры не любят ручной CSV)
- Конкурент с готовой Booksy-интеграцией может появиться раньше нас. **Митигация:** скорость остального продукта, и наш более глубокий учёт расходов и маржинальности (Booksy не делает)

### Что мониторим

- Сколько % юзеров используют Booksy (опрос в онбординге)
- Сколько % приходят к CSV-импорту в первую неделю (метрика "тяга к интеграции")
- Когда соберём 50+ платящих, делаем research-спайк и финализируем ADR
