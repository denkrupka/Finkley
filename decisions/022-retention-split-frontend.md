# ADR-022: Retention split (новые/постоянные) на клиенте, без RPC

**Дата:** 2026-05-22
**Статус:** Принято
**Контекст:** Коммит `3f83542`, `StaffPerformanceSection.tsx`

## Контекст

Юзер показал что общая колонка «Возвраты» в Reports → Мастера →
«Эффективность мастеров» — слепая. Из неё нельзя понять, теряем
ли мы новых клиентов (плохая отработка первой встречи) или
отваливается база постоянных (плохой follow-up).

Запрошено разделить на 2 метрики:

- **Возврат новых** — % клиентов с первым визитом у мастера в
  периоде, которые вернулись хотя бы один раз
- **Удержание постоянных** — % клиентов с ≥2 визитов всей истории
  (первый — ДО периода), пришедших в текущем периоде

## Проблема

Обе метрики требуют **полной истории визитов клиент-мастер** (для
определения «впервые»), не только окно периода. Текущий
`useStaffPerformanceAdvanced` RPC даёт только агрегаты за период.

Варианты:

1. **Новый RPC** `staff_retention_breakdown(salon, start, end)`:
   - Чистое серверное решение, скейлится для больших салонов
   - Минусы: миграция, SQL window-функции, тесты RPC
2. **Frontend lite-выборка всех visits**:
   - `select staff_id, client_id, visit_at where kind='visit'` —
     3 колонки × N rows
   - Для маленьких салонов (≤1000 визитов) = ~50 KB, ~50 ms compute
3. **Расширить существующий RPC** добавив поля retention_new,
   retention_regular:
   - Меньше изменений, но RPC становится сложным (3 запроса в одном)

## Решение

Выбран **вариант 2** (frontend lite-выборка) для текущей стадии.

```ts
const { data: allTimeVisits } = useQuery({
  queryKey: ['visits-lite-all-time', salonId],
  queryFn: async () =>
    supabase
      .from('visits')
      .select('staff_id, client_id, visit_at')
      .eq('salon_id', salonId)
      .eq('kind', 'visit'),
})
```

Дальше per-(staff,client) Map → first_at + count. Решение для каждого
мастера за O(N) где N = всего визитов салона.

## Обоснование

- **Простота**: не нужна миграция, нет нового RPC, нет дополнительной
  моверы (sql тест → unit test on backend).
- **Маленький bundle**: для 1000 визитов ≈ 50 KB через PostgREST.
- **Refetch cheap**: кэш на 5 мин, инвалидируется только новой
  записью визита.
- **Гибкость**: легко поменять порог «постоянный = ≥5» в одном месте,
  без миграции.

## Когда переходить на RPC

Триггеры пересмотра:

1. У какого-то салона >5000 визитов за год — выборка станет тяжёлой.
   Решение: добавить `useStaffRetentionAdvanced` RPC, переключить
   только этот компонент.
2. Появится несколько мест где нужна та же логика — DRY через RPC.
3. Если retention становится критичной метрикой для AI-аналитики —
   серверный compute точнее для batch-processing.

## Защита от регрессии

Нет dedicated unit-тестов — логика тривиальная (Map + filter), хорошо
видна в коде. Если эта секция станет более сложной, добавить shadow
test для функции `statsFor`.

## Альтернативы (рассмотренные)

- **Materialized view**: переусложнение для текущего масштаба.
- **Per-client counter в clients table** (denormalize): риск
  рассинхронизации после ручных удалений визитов.
