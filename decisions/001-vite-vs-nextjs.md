# ADR-001: Vite + React SPA вместо Next.js

## Статус

`Accepted` · 2026-05-05

## Контекст

Нужен фреймворк для фронтенда. Главные ограничения:

1. **Хостинг — только GitHub Pages.** У владельца нет VPS, есть только GitHub + Supabase. Это работающий стек на его другом продакшен-проекте (sasovsky).
2. **Бесплатно на старте.** GitHub Pages бесплатен.
3. **Простой стек для соло-разработчика.**

Next.js предполагает SSR/RSC, что требует Node.js runtime. На GitHub Pages не работает (только статика). `next export` — это всё равно SPA, но с лишней сложностью Next-абстракций.

## Решение

**Vite + React 18 + TypeScript + React Router v6.**

- Один статический бандл с GitHub Pages
- Клиентский роутинг через React Router
- Серверная логика — в Supabase Edge Functions (Deno)
- Авторизация — Supabase Auth (JWT в браузере)

Лендинг — отдельное приложение (Astro) в монорепе.

## Альтернативы

- **Next.js + Vercel** — отклонено: владелец явно отказался от Vercel.
- **Next.js static export на GitHub Pages** — отклонено: лишняя сложность Next.js без преимуществ.
- **Remix** — отклонено: требует Node.js runtime.
- **Astro для всего** — отклонено: Astro для контентных сайтов; для интерактивного SaaS React+Vite проще.

## Последствия

### Положительные

- 100% бесплатный хостинг
- Стек идентичен другому работающему проекту владельца
- Нет SSR-сложностей
- Быстрая сборка Vite

### Отрицательные

- Нет SSR — плохо для SEO. **Митигация:** лендинг отдельный на Astro, индексируется хорошо. SaaS-приложение SEO не нужно.
- Нет Server Components — больше JS бандла. **Митигация:** code splitting через React.lazy.
- Initial loading slower without SSR. **Митигация:** skeleton loading.

### Что мониторим

- Если бандл >500KB gzipped — пересмотреть code splitting
- Если SEO лендинга страдает — переосмыслить
