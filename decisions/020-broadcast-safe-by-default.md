# ADR-020: Broadcast prefs — safe-by-default (OFF)

**Дата:** 2026-05-22
**Статус:** Принято
**Контекст:** Миграция `20260521000023_broadcast_prefs_default_off.sql`

## Контекст

В таблице `salons` хранится `broadcast_prefs jsonb` — настройки рассылок
на 3 типа (marketing / visit_reminder / review_request), per-channel
(email / sms).

Изначально (миграция `20260521000022`) дефолт был **all-true** — все
каналы включены по умолчанию. Логика: «существующее поведение не должно
сломаться при выкатке миграции».

## Проблема

При первой реальной демонстрации пользователю выяснилось:

1. Юзер ожидает что новый салон **не должен** автоматически рассылать
   ничего клиентам до явного включения.
2. Cron'ы `client-overdue-push` (09:00 UTC) и `send-review-request`
   (каждые 6ч) при дефолте all-on могли уйти всем клиентам без явного
   согласия владельца — это juridical risk (GDPR + Polish telecom).

## Решение

Изменить дефолт на **all-false**:

- Миграция `20260521000023` обновляет default колонки + backfill всех
  существующих салонов на all-false.
- Серверный `getBroadcastChannels` использует `obj.email === true`
  (не `obj.email !== false`) — любой невалидный/отсутствующий ключ
  даёт false.
- Клиентский `normalize` в `useBroadcastPrefs` тоже.

## Последствия

- **Breaking для проды:** все 3 существующих salonа перешли в state
  «ничего не шлёт». Юзер должен явно включить в `/marketing`.
- **Юридически безопасно:** клиенту нельзя слать без согласия владельца,
  владелец нельзя пожаловаться на «неожиданные SMS-расходы».
- **UX-новизна:** при создании нового салона юзер должен зайти в
  `/marketing` после онбординга и включить нужное. Tutorial в онбординге
  это упоминает.

## Защита от регрессии

Unit-тесты `apps/web/src/lib/broadcast-prefs-normalize.test.ts` — 7 кейсов
проверяют:

- null/undefined → all OFF
- empty object → all OFF
- невалидные типы (string вместо bool, array вместо object) → OFF
- частичный prefs (только email задан) → sms остаётся OFF
- лишние kinds игнорируются без падения

## Альтернативы (отвергнутые)

1. **Дефолт all-on + флаг `auto_enable=false` на новых салонах** —
   усложняет логику, разные дефолты для старых vs новых.
2. **Email default-on, SMS default-off** — частично решает risk, но
   email тоже без consent шлётся → лучше единообразное OFF.
3. **Отдельный onboarding-шаг «согласись на рассылки»** — навязчиво,
   юзер хочет настроить позже. Tutorial в `/marketing` достаточно.

## Что нужно сделать дальше

- Документацию для пользователей: после онбординга показывать
  notification «Включить рассылки в /marketing» (низкий приоритет).
- Аналитика: метрика «% салонов с хотя бы одним включённым каналом» —
  если низкая, пересмотреть подсказки.
